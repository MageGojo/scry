# 设计 · 扩展系统(Extender / Extensions)

> 目标:给 scry 补上对标 Burp **Extender** / mitmproxy **addons** 的扩展能力。
> 一句话设计:**定义唯一的「Hook 契约 + 宿主回调 API」,宿主只认一个 `ExtensionHost` 抽象;
> 语言差异下沉到「Runner」** —— Rust 走 native dylib 或 WASM,Python 走外部进程 RPC。
> 内核(`scry_proxy` MITM)零改动,只在三个接缝挂钩子。

---

## 1. 为什么这样设计(约束驱动)

- **边界类型现成**:`scry_core::HttpFlow` 本身 `Serialize/Deserialize`,跨进程 / 跨 WASM 零成本传。
- **接缝唯一**:抓包内核就一条线(`scry_proxy`):读请求 → 转发 → 读响应 → 落盘 → 回传。
  钩子只挂这三处(下 §6),不污染业务逻辑。
- **已有「事实扩展」**:`scry_scan`(扫描)、`scry_codec`(解码变换)、`scry_seq`、`scry_diff`、
  `scry_analyze`(curl/python/js 代码生成)都是纯函数 crate —— 它们是验证扩展 API 的现成参照样本,
  将来可改写成「第一方扩展」反过来吃自己的 API。
- **零环境出包硬约束**:Windows/macOS 要双击即用。⇒ **不嵌 CPython**(打包太重 + GIL 卡 UI),
  Python 走外部进程(用户自带 `python3`);Rust 扩展走 WASM(wasmtime 静态链接)或 dylib(无额外运行时)。
- **绝不因扩展卡死实时流量**:被动钩子离开热路径;内联钩子带超时 + fail-open。

---

## 2. 总体架构:一个契约,三种运行时

```
                         ┌──────── scry_ext_api (新 crate,只依赖 scry_core) ────────┐
   scry_proxy 三个接缝 ─▶ │  trait ExtensionHost: on_request / on_response /          │
   (cfg.hooks: 可选)      │                        on_flow_complete                   │
   scry_app 持有 Registry │  trait Extension(单个扩展) + HostServices(回调) + Manifest │
                          └───────────────────────────┬──────────────────────────────┘
                                                       │ 同一契约,多种 Runner
                ┌──────────────────────┬───────────────┼────────────────────────┐
                ▼                      ▼                ▼                        ▼
        ⓪ Builtin(进程内)     ① Native dylib       ② WASM(sandbox)      ③ Process(进程外)
        第一方,直接 impl       libloading + cdylib   wasmtime + WIT        spawn + JSON/msgpack RPC
        scry_scan 等            &mut HttpFlow 零拷贝  序列化边界           序列化边界(stdio/uds)
        最快·随包               最快·能崩·仅可信     安全·热重载·任意语言  崩溃隔离·任意语言(Python⭐)
```

- **`scry_ext_api`**:纯类型 + trait,无 IO。是契约的唯一真相源(含 ABI 版本号)。
- **`ScryApp` 内的 `ExtRegistry`**:加载 / 启停扩展,实现 `ExtensionHost`,按加载顺序 fan-out 到各扩展;
  把扩展产生的日志 / findings 收进线程安全缓冲(供 UI 读)。`Arc<ExtRegistry>` 注入 `ProxyConfig.hooks`。
- **`scry_proxy`**:只依赖 `scry_ext_api`,在三个接缝调用 `cfg.hooks`(`None` = 完全无行为变化)。

### 各运行时取舍

| 运行时 | 语言 | 速度 | 隔离/安全 | 热重载 | 随包 | 定位 |
|---|---|---|---|---|---|---|
| ⓪ Builtin | Rust | ★★★ | 同进程(可崩) | 否 | 是 | 第一方能力(scan/codec…) |
| ① Native dylib | Rust(C-ABI) | ★★★ | 同进程(可崩) | 半 | 否 | 可信高性能扩展 |
| ② WASM | Rust/TinyGo/JS/(py) | ★★ | 沙箱(强) | 是 | 是 | **第三方默认推荐** |
| ③ Process | Python/Node/任意 | ★(IPC) | OS 进程(强) | 是 | 否(自带运行时) | **Python⭐ / 脚本党** |

- **Rust 扩展**:图省事/安全 → ②WASM(`wasm32-wasip2`);图极限性能 + 可信 → ①dylib(`scry_plugin_entry()` C-ABI + ABI 版本校验,`libloading` 加载)。
- **Python 扩展**:③外部进程 + stdio JSON-RPC。崩溃隔离、零打包负担、同协议顺带支持 Node/Ruby/任意二进制。
  进阶可用 `componentize-py` 把 Python 也编成 ②的 WASM 组件(后话)。

## 3. Hook 契约(核心)

照搬 Burp Montoya + mitmproxy addon,映射到 scry 管线。

### 3.1 流量钩子
```rust
pub enum HookAction {
    Continue,                  // 放行(flow 可能已被原地修改)
    Drop,                      // 丢弃:不转发,直接断
    Respond(SynthResponse),    // 短路:由扩展直接给客户端造一个响应(Match&Replace / mock)
    Pause,                     // 转人工拦截队列(对接将来的 Intercept 断点)
}

pub trait Extension: Send + Sync {
    fn manifest(&self) -> &ExtManifest;
    // 内联:转发前,可改 method/path/headers/body
    fn on_request(&self, _f: &mut HttpFlow, _h: &mut dyn HostServices) -> HookAction { HookAction::Continue }
    // 内联:回传前,可改 status/headers/body(响应体改写见 §6 注意)
    fn on_response(&self, _f: &mut HttpFlow, _h: &mut dyn HostServices) -> HookAction { HookAction::Continue }
    // 被动:落盘后只读副本,给扫描/索引/日志(绝不阻塞流量)
    fn on_flow_complete(&self, _f: &HttpFlow, _h: &mut dyn HostServices) {}
}
```

### 3.2 工具扩展点(喂给已有页签;P3/P4 接)
- `register_scanner_check`(被动 + 主动)→ Scanner 页(`scry_scan` 模型)。
- `register_transform` → Decoder 页(`scry_codec` 模型);`register_payload_generator/processor` → Intruder。
- `register_context_menu_item` → Proxy/Repeater 右键(复用 `chrome.rs::context_menu_overlay`)。
- `register_message_editor_tab` → 自定义请求/响应查看 Tab。
- `register_session_handler` → 发包前自动重认证 / 跑宏(Burp session handling)。

### 3.3 宿主回调 `HostServices`(扩展能调 scry 的能力)
```rust
pub trait HostServices {
    fn log(&mut self, level: LogLevel, msg: &str);     // → Logger 页
    fn emit_finding(&mut self, f: Finding);            // → Scanner
    fn get_setting(&self, key: &str) -> Option<String>;// 扩展私有设置(来自 manifest settings_schema)
    fn set_kv(&mut self, key: &str, val: &str);        // 扩展私有持久化
    // P1+:fn send_request(&mut self, req: HttpRequest) -> HttpResponse;  // 复用 scry_proxy::replay
    // P1+:fn highlight(&mut self, color); fn comment(&mut self, text);    // Burp 注释
}
```
> 关键:`HostServices` 必须能从 **proxy 线程**调用(代理跑在独立 tokio 线程)。因此其实现是线程安全的
> 事件汇聚(`Arc<Mutex<…>>` / channel),由 UI 主线程每帧 / 每 tick 排空,再 `cx.notify()` 刷新。

## 4. 数据边界(序列化)
- **②WASM / ③Process**:`HttpFlow` 走 JSON(body 用 base64)或 msgpack(body 原始字节)。带 `schema_version`。
- **⓪Builtin / ①Native**:直接传 `&mut HttpFlow`(零拷贝,最快)。
- 同一份 `HttpFlow` 类型,序列化只是不同运输层 —— 契约不变。

## 5. Manifest + 自动设置 UI(借鉴 sing-box 插件)

每个扩展带 `manifest`(对标 GUI.for.SingBox 的 `manifest.yaml`,用户已熟):
```yaml
id: passive-secret-scan
name: 被动密钥扫描
version: 0.1.0
author: scry
kind: builtin | native | wasm | process
entry: ./libfoo.dylib | foo.wasm | "python3 foo.py"
abi: 1
permissions: [traffic.read, traffic.modify, net.outbound, storage, "fs.read:~/.scry"]
hooks: [on_flow_complete, scanner_check]
settings_schema:                 # 像 singbox 的 Input/Switch,自动渲染成设置表单
  - { key: min_len, kind: number, label: 最小长度, default: 16 }
  - { key: include_cookies, kind: switch, label: 扫描 Cookie, default: true }
```
- `settings_schema` → Extender 页**自动生成配置表单**(singbox 那套组件驱动 UI 的玩法)。
- `permissions` → 安装时按权限提示授权(`traffic.modify` / `net` / `fs` 高危项明确告知)。
- 灵感落地:mitm-core 的 `<URL_Regex> -> redirect url:…` 重写 DSL → 做成第一方 **Match&Replace 扩展**,
  正好补上进度表「拦截改包」方向。

## 6. 与现有管线的接缝(`scry_proxy`)

抓包内核就两个落盘点,两处对称挂钩:
- `crates/scry_proxy/src/lib.rs::proxy_plain`(明文 HTTP:绝对 URI + 隧道内 origin-form 共用)。
- `crates/scry_proxy/src/mitm.rs::intercept_https`(HTTPS 解密)。

挂法:
1. **on_request**:收齐请求(method/path/headers/body)后、`build_origin_request` 转发**之前**调用。
   `Drop`→直接返回断连;`Respond`→给客户端写自造响应并返回;`Continue`→用(可能改过的)请求部件转发。
   → 请求侧本就要重建 origin-form,改写天然干净,**P0 即支持**。
2. **on_response**:读到响应、回传客户端**之前**调用。
   - ⚠️ 响应体改写要处理 `Content-Encoding`/`Transfer-Encoding` 重编码,易踩坑;
     **P0 只挂 on_request + on_flow_complete**,on_response 的 body 改写留 **P1**(在 `mitm` 路径上做,
     因为那里 `read_message` 已 dechunk,body 干净,重编码安全)。
3. **on_flow_complete**:`save_flow` **之后**对副本调用(被动只读)。零风险,P0 即用。

`ProxyConfig` 加一个可选字段(默认 `None` = 零行为变化):
```rust
pub struct ProxyConfig { /* … */ pub hooks: Option<ExtHooks> }
// ExtHooks = newtype(Arc<dyn ExtensionHost>) + 手写 Debug,保住 ProxyConfig 的 #[derive(Debug, Clone)]
```

### 防回环 / 安全
- 扩展跑在 scry 进程内(⓪①②)或子进程(③),都不改系统代理 → 不影响 §抓包方案重构 的回流设计。
- ③Process 扩展若用 `net.outbound`,经 `HostServices::send_request` 走 scry 自己的 replay/upstream,
  不直接拿 socket → 仍受统一回流约束。

## 8. 安全 / 性能(硬规矩)
- **被动钩子离开热路径**:clone flow → 异步队列 → fan-out,不挡 save-first、不挡回写。
- **内联钩子超时 + fail-open**:③Process 的 `on_request` 超 N ms → 原样放行 + 告警,坏扩展绝不卡死流量。
- **三档信任**:②WASM=默认沙箱(不可信也敢装)/ ③Process=OS 隔离 / ①Native=仅可信(全权、能崩)。
- **顺序确定**:按加载顺序 / 优先级跑;多个扩展改同一字段以最后一个为准(日志可见)。
- **零环境出包不破坏**:wasmtime 静态链接、dylib 无额外运行时;Python 子进程 = CPython 不进包。

## 9. 落地分期
- **P0(本阶段)**:`scry_ext_api` crate(契约 + manifest 类型 + 单测)；`scry_proxy` 接 **on_request + on_flow_complete**(默认 no-op,additive)；`scry_app` 加 `ExtRegistry` + Extender 页 + 两个 builtin 演示扩展(流量计数 / 被动密钥扫描),`capture.rs` 注入 hooks。→ 端到端跑通「扩展能看见 / 标记实时流量」。
- **P1**:③外部进程 Runner(**Python 优先**)+ stdio JSON-RPC + 超时/fail-open + 示例 Python 扩展;补 `on_response` body 改写(mitm 路径)与 `HostServices::send_request`。
- **P2 ✓(已完成)**:②WASM Runner —— `scry_ext_host::wasm`(`WasmExt`)。选 **core module + 线性内存 JSON ABI**(而非 WIT/Component:测试可内联 WAT、免 `wit-bindgen` 工具链,且与进程 Runner 的 JSON 协议同构)。**无 import 函数**(纯沙箱)+ **fuel** 防死循环 + **StoreLimits 64MB** 防内存炸弹 + **每次钩子新建实例**(强隔离 + `&self` 并发无锁)。reply / manifest 协议抽到 `wire.rs` 与进程 Runner 共用;`discover` 按 `manifest.json` 的 `wasm` 字段分流。示例 `extensions/wasm-demo`(Rust→wasm32,76KB)。
- **P3**:①Native dylib Runner;让第一方 crate(scry_scan 等)以扩展形态加载。
- **P4**:manifest 驱动设置 UI 完整化 + 右键/消息编辑器/会话处理器扩展点 + Plugin-Hub 式仓库。

## 10.5 P1 落地状态(外部进程 / Python Runner)

- ✅ **新 crate `scry_ext_host`**(不依赖 gpui,可独立单测):`ProcessExt` 实现 `Extension`,
  stdio **JSONL RPC** + 后台读线程 + `recv_timeout` 超时 + **fail-open**(超时/出错按 `Continue` 放行)+
  进程退出 `Drop` 收子进程。`manifest` 握手让扩展自述(name/version/hooks/permissions,校验 ABI)。
- ✅ **发现机制**:`discover(dir)` 扫描子目录里的 `manifest.json`(`{ "command": ["python3","main.py"], "timeout_ms": 1500 }`)。
  默认目录 `~/.scry/extensions/`(不存在则不加载任何进程扩展 = 零额外进程)。
- ✅ **示例 Python 扩展**:`scry/extensions/py-demo/`(`main.py` 纯标准库:on_request 加 `X-Scry-Py` 头 / on_flow_complete 对 4xx·5xx 报 finding)。
  **安装**:`cp -r scry/extensions/py-demo ~/.scry/extensions/`,重启 scry 即在 Extender 页看到「Python 演示扩展(进程)」。
- ✅ **接入 UI**:`scry_app::ext::ExtRegistry::with_builtins()` 启动时 `discover` 并把进程扩展并入注册表(加载成功/失败写进 Extender 近期活动)。
- ✅ **验证**:`scry_ext_host` 集成测试用真实 `python3` 拉起示例,断言握手 + on_request 改写回灌 + 5xx finding(无 python3 优雅跳过)。
- ✅ **P1.5① on_response 响应改写**:`scry_proxy::mitm` 接 `on_response`(`Drop`/`Respond`/`Continue` 改写)。
  `ExtensionHost::wants_response_hook()` 门控:无声明 `on_response` 的扩展时,代理按原貌转发 `raw`(**零保真损失**);
  有则在 dechunk 后用 `build_http_response` 重建(丢 chunked 框架 / 重算 `Content-Length` / 强制 close)。落盘存改写后内容。+3 单测(scry_proxy 26 测过)。
- ✅ **P1.5② `HostServices::send_request`**(扩展主动发包,做主动扫描 / 取数据):
  - 契约加 `ExtRequest{method,url,headers,body}` / `ExtResponse{status,headers,body,error}` + `HostServices::send_request`(默认「不支持」)。
  - `scry_proxy::replay::ReplayRequest::from_url`(解析完整 URL);`scry_app::Collector::send_request` 复用 `replay::send`,
    与抓包**同上游**(`ExtRegistry::set_upstream` 在抓包启动时注入),在抓包多线程 runtime 上 `spawn`+`recv_timeout` 同步取结果。
  - **进程扩展双向 RPC**:扩展在钩子内发 `{"id":N,"call":"send_request","request":{…}}`,宿主代为执行并回 `{"id":N,"result":{…ExtResponse…}}`,
    然后继续等扩展的最终回复。示例 `py-demo` 对 `/active-probe` 路径主动发包(集成测试验证回环)。

### 进程扩展双向 RPC 线路(在一次钩子内,扩展→宿主)
```
ext → host:  {"id":7,"call":"send_request","request":{"method":"GET","url":"https://example.com/"}}
host → ext:  {"id":7,"result":{"status":200,"headers":[...],"body":[...]}}
ext → host:  {"id":2,"action":"continue","findings":[...]}   # 同一钩子的最终回复
```

线路协议(host→ext / ext→host,每行一条 JSON):
```
{"id":2,"hook":"on_request","flow":{...HttpFlow...}}
{"id":2,"action":"continue","flow":{...改写后...},"logs":[{"level":"Info","msg":"…"}],"findings":[{...}]}
```
`action ∈ continue|drop|respond|pause`;`respond` 附 `"response":{status,headers,body}`;`severity` 用 `Info|Low|Medium|High|Critical`。

## 10. 取舍与边界(诚实记录)
- ⓪①与 scry 同进程 → 崩溃会带崩 scry;②③隔离更安全但有(序列化/IPC)开销。
- on_response 响应体改写涉及重编码,P0 不做(只读 / 仅请求改写),P1 在 dechunk 后的 mitm 路径补。
- Python 选「外部进程」而非「嵌入(pyo3)」:换取崩溃隔离 + 零打包负担,代价是每次内联钩子一次 IPC 往返
  (被动钩子无所谓;内联钩子用超时 + fail-open 兜底,鼓励能被动就被动)。
