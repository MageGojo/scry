# 设计 · 抓包源选择(Wireshark 式)+ 默认代理 + Proxifier 配合

> ⚠️ **已被 [`设计-抓包方案重构.md`](设计-抓包方案重构.md) 取代(superseded)。**
> 本文描述的是旧的「抓包源 = 代理 + 各网卡」扁平列表模型(按机制组织)。新方案改为「**抓包目标**」四卡门面
> (T1 内置浏览器 / T2 托管程序 / T3 整机透明·嗅探 / T4 对接代理客户端),以 scry 自管流量源为零配置默认。
> 本文保留作**机制细节 / 后端分发(`capture.rs` 的 `capture_mode`+`iface_sel`)参考**;「网卡列表 + 火花线」并入新方案的「抓整机」卡。

## 背景需求(本轮三点)
1. **默认用代理模式**:打开即以 MITM 代理(`127.0.0.1:8888`)为默认抓包源,能解密 HTTPS。
2. **配合 Proxifier**:系统代理被 Quantumult X / Surge 占着时,用 Proxifier **按进程**把目标流量喂给 scry(这正是 scry「不抢系统代理」的原始设计)。
3. **Wireshark 式抓包源点选**:在仪表盘做一个「抓包源」列表,可点选要抓哪个、有默认项,点「开始抓包」就抓选中的那个。

## 核心决策:把两种抓包方式收敛成一个「抓包源」扁平列表
仪表盘(`Dashboard` 页)用一个**可点选的扁平列表**表示所有抓包源:
- **第 0 项 · MITM 代理**(`127.0.0.1:8888`)—— **默认源**。可解密 HTTPS;配合 Proxifier / 系统代理路由。
- **第 1..N 项 · 各网卡**(`utunN` / `en0` / `lo0` …)—— 内核被动嗅探(类 Wireshark),HTTP 明文可解析,HTTPS 仅 SNI。列表首个网卡(默认路由网卡)标「默认网卡」。

点选某项 = 设置抓包源,**复用既有后端分发**(`capture.rs`),不改抓包引擎:
- 选「代理」→ `capture_mode = Proxy`。
- 选「网卡 i」→ `capture_mode = Kernel` + `iface_sel = i`。

下标映射:
- `current_source_index()`:`Proxy → 0`;`Kernel → 1 + iface_sel`。
- `select_source(idx)`:`0 → Proxy`;`idx≥1 → Kernel + iface_sel = idx-1`。

## 交互
- 启动落 `Dashboard` 页(先选源、再开抓,贴合 Wireshark「先选网卡」习惯)。默认选中代理。
- 选中行:左侧高亮条 + 玻璃底 + 主色描边 + 右侧勾选图标。
- 每行右侧有一条「火花线」(**纯装饰**,模拟 Wireshark 网卡旁的流量小图;固定高度序列,非实时速率,避免误导;仅在「选中且抓包中」点亮成功色)。
- **抓包进行中禁止切源**(点其它源给提示「请先停止抓包」),避免热切换引擎 / 反复弹授权。
- 顶部:页标题 + 当前源摘要 + 大「开始 / 停止抓包」按钮;下方概览指标(总流量 / HTTPS / HTTP / 运行时长)。
- 底部提示卡随当前源切换:
  - 代理源 → 三条:把目标 App/浏览器代理指向 `127.0.0.1:8888`;Proxifier 按进程转发 + 防回环;安装信任根证书(带「打开证书设置」按钮)。
  - 网卡源 → 被动嗅探无需代理 + HTTPS 不解密(仅 SNI)+「授权抓包(BPF)」按钮。

## Proxifier 配合(防回环要点)
- 目标 App 不直接设代理时,用 **Proxifier 进程规则**:把目标进程的流量代理到 `127.0.0.1:8888`(scry)。
- **必须给 scry 自身(及 Quantumult X / Surge 客户端进程)加 Direct 规则**,否则 scry 把解密后的上游请求再交回 Proxifier → 又被转回 scry → 死循环。
- 链路:目标 App →(Proxifier 按进程)→ scry 解密 →(scry 自身 Direct,走系统默认路由)→ QX/Surge → 出网。系统代理 / QX 一行不动,满足「不抢代理」。

## 取舍
- 火花线是装饰、非真实速率(没有每网卡实时字节统计,造假会误导)。
- 抓包中不支持热切源(MVP,换源需先停)。
- 抓包源不显示真实 IP / 速率,仅显示能力标签(解密 / 仅 SNI)与默认徽标。
