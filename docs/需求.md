# Scry 需求文档

## 背景
- 需要一款**像 Burp Suite 一样**的安全渗透 / 流量分析工具,用于抓包、解密、改包、重放、扫描。
- 已有成熟基建:同级 [`mage-ui`](../../mage-ui) —— 基于 gpui 的自研组件库(主题 + 虚拟化 Table +
  侧边栏 + 输入 + 图标等),且 vendor 了可改造的 gpui 源码。新工具 UI 直接复用并反哺它。
- 已有 `burp-capture` 工作流:**Proxifier 路由 + 代理解密分析**。Scry 延续此思路。

## 目标
1. 纯 Rust 内核(代理 / 证书 / 存储 / 扫描),`mage_ui`(gpui)原生界面,**优先 macOS(arm64)**。
2. **开箱即用、不抢系统代理**:默认由 **scry 自己掌控流量源的启动**(内置浏览器 / 托管启动目标程序,自动喂代理 + 注入 CA),
   零配置即可解密;**配合代理客户端(sing-box / QX / Proxifier)为进阶**,用于抓「已运行的任意软件」。详见 [`设计-抓包方案重构.md`](设计-抓包方案重构.md)。
3. HTTP/S **MITM**:解密、查看、改写、重放;请求 / 响应**先落盘后分析**(SQLite + 去重)。
4. Burp 式工作台:Proxy(history + 拦截改包)/ Repeater(重放)/ Intruder(爆破)/ Scanner(扫描)。
5. UI 全量复用 `mage_ui`;缺的组件补进 `mage_ui`(反哺)。

## 平台
- 优先 **macOS(Apple Silicon / arm64)**;Linux / Windows 后续评估。

## 关键决策(已确认)
- 名称:**Scry**(占卜窥视 = 截获流量;与 `mage_ui` 同属法师世界观,组成「法师 + 占卜术」产品线)。
- **不内置 Proxifier 式进程劫持**(规避 macOS Network Extension 签名)。
- **[重构]抓包模型 = 「按抓什么」+ scry 自管流量源**:默认零配置(内置浏览器 / 托管启动);
  对接代理客户端(sing-box / QX / Proxifier)、透明、嗅探为进阶。引擎(MITM + upstream)零改动复用。详见 [`设计-抓包方案重构.md`](设计-抓包方案重构.md)。
- UI = `mage_ui`,顺路完善 `mage_ui` 组件库。

## 范围
### MVP(阶段 0/1)
- gpui 窗口跑通(复用 `mage_ui`),Burp 式三栏外壳 + 流量列表(虚拟化 Table)。
- 代理 MVP:绑定端口,接受 Proxifier 路由来的连接;CONNECT 隧道(HTTPS 暂透传)+ 明文 HTTP 抓取并落盘。
- CA:生成根证书(供后续 TLS MITM 解密)。
- 存储:SQLite 落盘 + 去重(请求先保存)。

### 暂不在 MVP(后续阶段)
- TLS MITM 真解密(用 CA 动态签发叶子证书,解密 HTTPS 明文)。
- 拦截改包(断点)/ Repeater / Intruder / Scanner。
- TLS 指纹伪装、底层 L2/L3 抓包(Wireshark 式)、零环境出包。

## 非目标
- 不做内核驱动 / 系统扩展级别的进程劫持(默认靠 scry 自管流量源;抓已运行的任意软件交给代理客户端 / 透明)。
- 不替代 Wireshark 的底层嗅探(已做被动嗅探 + pcapng 导出,作为「仅元数据」的降级路径)。

## 待确认(open questions)
- 默认代理端口(暂定 `127.0.0.1:8888`)。
- 证书 / 落盘目录(暂定 `~/.scry/`)。
- 拦截改包的 UI 交互形态(队列式 / 单步放行)。
