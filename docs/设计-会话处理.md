# 设计 · 会话处理 + 登录宏(scry_session)

> 第一档 ROI 缺口 #3:**Session / Macros 自动重登 + CSRF/JWT 续期**(authenticated 扫描刚需)。
> 对标 Burp **Session Handling Rules + Macros** / Caido Authentication。

## 1. 解决什么

authenticated 扫描(SQLi/XSS/Nuclei…)发大量请求,中途会话(Cookie/JWT)过期 → 后续请求全部跳登录 / 401
→ 误判「无漏洞」。对策:
1. **登录宏**:一条能重建有效会话的请求(`POST /login` 拿新 Cookie;或 GET 页面取新 CSRF)。
2. **捕获**:从宏响应抽会话 —— `Set-Cookie`(→ Cookie 头)+ 可选令牌(正则,CSRF/JWT)。
3. **注入**:把会话套到后续每个扫描请求(合并 Cookie 头 / 注入令牌头 / `{{token}}` 占位替换)。
4. **登出检测 + 自动重登**:响应像「掉登录」(401/403、重定向 login、命中正文标记)→ 重跑登录宏刷新会话。

## 2. 架构(契合现有范式)

- **纯函数内核 crate `scry_session`**(零 IO,可单测):
  - `lib.rs`:`SessionState`(cookies + token)、`CaptureSpec`/`ApplySpec`/`LoggedOutSpec`、
    `cookie_header` / `substitute`(`{{token}}`/`{{csrf}}`)/ `inject_headers`(合并 Cookie + 注入令牌头)。
  - `capture.rs`:`parse_set_cookie` / `extract_token`(正则,body/header)/ `build_session`。
  - `detect.rs`:`looks_logged_out`(状态码 / 重定向 login / 正文标记)。
- **`scry_app/src/session.rs`**(IO + 页面):
  - `SessionPlan`(登录宏 `ReplayRequest` + 捕获/注入/登出规则 + 上游)。
  - `run_login_macro(plan)`:复用 `scry_proxy::replay::send` 跑宏 → `build_session` 捕获会话。
  - `apply_session_to(headers, body, st, apply)`:注入 Cookie/令牌头 + `{{token}}` 替换。
  - `Tab::Session` 页:登录宏编辑(目标 + 原始请求)→「运行登录宏」(replay 跑宏,展示捕获的
    会话/状态)→ 捕获/注入配置(Set-Cookie 开关、令牌正则、令牌注入头、掉登录正文标记)→ 活动会话卡。
  - `session_plan(cx)`:仅当「套用到扫描」总开关开启且宏有效时返回 `Some`,供各 runner 取用。

## 3. 扫描器集成

各 runner 在 `start_*` 取 `self.session_plan(cx)` 传入后台 runner:
- **Nuclei**(完整):开扫前 `run_login_macro` 建会话;每请求 `apply_session_to` 注入;
  发完检测 `looks_logged_out` → 中途**自动重登**(克隆 plan 跑宏刷新 → 重发一次)。
- **SQLi / XSS(反射)**:开扫前建会话 + 注入基准请求(所有探测变异自动带上会话)。

> 引擎零改动(`scry_proxy`/`scry_sqli`/`scry_xss`/`scry_nuclei` 不动);集成全在 app 层 runner。

## 4. 子集边界(诚实声明)

- 登录宏目前是**单请求**(够覆盖绝大多数 form/JSON 登录);多步宏(先取 CSRF 再登录)后续可扩。
- 中途自动重登目前在 **Nuclei**(单发送循环,接入干净);SQLi/XSS 为**开扫前建会话**(覆盖 80%
  场景:会话在扫描前过期/未建立);其内部多发包点的逐请求重登留作后续。
- 存储型/DOM XSS(浏览器驱动)用浏览器自身 Cookie,未走本会话注入。
