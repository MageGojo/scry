# Scry 设计与关键决策

> **抓包方案总纲见 [`设计-抓包方案重构.md`](设计-抓包方案重构.md)**(canonical)。
> 模型已从「按机制(代理 / 各网卡)」重构为「**按抓什么**」+ scry 自管流量源(内置浏览器 / 托管启动为零配置默认,
> 对接代理客户端 / 透明 / 嗅探为进阶)。本文以下「配合 Proxifier」等描述为**机制细节 / 进阶路径**,不再是默认门面。

## 总体架构
```
                    ┌─────────────────────────────────────────┐
   目标 App ──┐     │ Scry 进程                                  │
            (Proxifier 路由)                                     │
   目标 App ──┼──▶ 127.0.0.1:8888 ──▶ scry_proxy (tokio)        │
            (按规则)  │      │  抓到即落盘(save-first)          │
   目标 App ──┘     │      ▼                                    │
                    │  scry_storage (SQLite + 去重)             │
                    │      │   ┌── scry_ca(rcgen 签发叶子证书)  │
                    │      ▼   ▼                                │
                    │  scry_core(共享类型 HttpFlow)            │
                    │      │  (channel)                        │
                    │      ▼                                    │
                    │  scry_app (gpui / mage_ui 界面)           │
                    └─────────────────────────────────────────┘
```
- **进程模型**:代理跑在后台 tokio 运行时(独立线程),UI 跑 gpui 主线程;两者用 channel 通信
  (代理把新 `HttpFlow` 推给 UI,UI 把改包 / 重放指令发给代理)。
- **流量从哪来(抓包目标,详见重构总纲)**:Scry 引擎只管「监听端口 + 解密 + 落盘」,谁把流量送进来不关心。
  - **默认 / 零配置**:scry **自己拉起流量源**(T1 内置浏览器 / T2 托管启动程序),并自动喂代理参数 + 注入 CA 信任。
  - **进阶 / 有外部依赖**:对接代理客户端(T4:sing-box 插件 / QX / Proxifier)、透明代理(T3:pf rdr)、被动嗅探(网卡)。
- **统一回流(upstream)**:scry 解密后经 `scry_proxy::upstream` 把到目标的连接交回 QX / sing-box 出网(墙内),
  目标交上游远程解析 → 根治「直连出不去」+「fake-ip 丢真实 SNI」。

## 目录结构
```
scry/
├─ Cargo.toml            # 虚拟 workspace:members = crates/*
├─ docs/                 # 需求 / 设计 / 进度
├─ crates/
│  ├─ scry_core/         # 共享类型(无重依赖):HttpFlow / Header / 指纹
│  ├─ scry_ca/           # CA:根证书生成 + 叶子证书签发(rcgen)+ 按域名缓存
│  ├─ scry_storage/      # SQLite 落盘 + 去重(save-first 铁律)
│  ├─ scry_decode/       # 展示用解码:Content-Encoding 解压 + charset + 分类(纯函数)
│  ├─ scry_analyze/      # 分析层:参数/Cookie/摘要提取 + 过滤搜索 + 导出 curl(纯函数)
│  ├─ scry_proxy/        # HTTP/S MITM 引擎 + 代理入口 + 透明抓包(pf,macOS)+ Repeater 重放(replay)
│  └─ scry_app/          # gpui 界面(mage_ui):Burp 式三栏
```

## 依赖关系
```
scry_app    →  mage_ui (path: ../mage-ui/crates/mage_ui)  →  gpui (mage-ui 自带 vendor)
            →  scry_core / scry_storage / scry_proxy / scry_decode / scry_analyze
scry_proxy  →  scry_core / scry_storage / scry_ca / scry_decode(replay)  (+ libc:macOS 透明抓包 pf)
scry_analyze→  scry_core / scry_decode —— 纯函数分析,不碰 IO/网络
scry_storage→  scry_core
scry_decode →  (flate2 / brotli / encoding_rs / serde_json) —— 不依赖其它内部 crate,保持轻量
scry_ca     →  (rcgen)
scry_core   →  (serde / sha1)
```

> 注:`scry_proxy::replay`(Repeater 重放)复用同 crate 的 `mitm::read_message`(HTTP/1.1 读)+
> `build_origin_request`(请求构造)+ rustls 客户端配置,不另起 TLS / 解析轮子;
> `scry_app` 接 `scry_analyze` 做 history 过滤 / 复制为 curl、接 `replay` 做 Repeater 面板(待落地)。

## 关键决策

### UI 复用 mage_ui(跨 workspace path 依赖)
- `scry_app` 只 path 依赖 `mage_ui`;`mage_ui` 已 `pub use gpui` 并在 `prelude` 导出所有 gpui 类型,
  所以 `scry_app` **无需直接依赖 gpui**,`use mage_ui::prelude::*;` 即可。
- `mage_ui` 的 `gpui = { workspace = true }` 由 **mage-ui 自己的 workspace** 解析到
  `vendor/zed/crates/gpui`(相对 mage-ui 根),跨 workspace path 依赖会正确就近解析。
- 工具链:默认 **rustc 1.96.0**(mage-ui 根无 `rust-toolchain.toml`,即用默认构建),Scry 同样,无需额外 pin。
- Metal:沿用 mage_ui 已开启的 `runtime_shaders`(运行时编译着色器,免 Metal Toolchain)。

### 代理(scry_proxy)—— MVP 范围与演进
- **MVP(本阶段)**:
  - `TcpListener` 绑定 `127.0.0.1:8888`,每连接一个 task。
  - 读首部用 `httparse` 解析请求行 + 头。
  - `CONNECT`:回 `200 Connection Established`,连到目标,`copy_bidirectional` 双向透传(HTTPS 暂不解密)。
  - 明文 HTTP(绝对 URI 请求):连目标 → 转发(强制 `Connection: close` 逼上游 EOF)→ 读完整响应 →
    组 `HttpFlow` → **落盘** → 回传客户端。每连接一请求(keep-alive 后续做)。
- **下一步(TLS MITM 解密)**:`CONNECT` 后不透传,而是用 `scry_ca` **动态签发目标域名叶子证书**,
  本地用 `tokio-rustls` 作 TLS 服务端与客户端握手,拿到明文再解析 / 落盘 / 重放。

### 证书(scry_ca)
- `rcgen` 生成**根 CA**(自签,长效),存 `~/.scry/ca.pem` / `ca.key`。
- 叶子证书按访问的域名**动态签发**(用根 CA 签),供 TLS MITM。用户需把根 CA 导入系统信任(同 Burp/Proxifier 流程)。

### 存储(scry_storage)—— 请求先保存铁律
- SQLite(`rusqlite` bundled,免系统 sqlite),表 `flows`,字段含
  `ts/method/scheme/host/port/path/url/req_headers/req_body/status/resp_headers/resp_body/duration_ms/fp`。
- **去重指纹** `fp = sha1(method | url | sha1(body))`,`fp` 唯一索引,`INSERT OR IGNORE` 天然去重。
- **save-first**:代理抓到 `HttpFlow` 立刻 `insert`,再推 UI / 分析。即使 UI / 进程崩溃也不丢。
- 落盘位置:`~/.scry/scry.sqlite`。

### 界面(scry_app)—— Burp 式三栏(复用 mage_ui)
- 左:`Sidebar`(Proxy / Repeater / Intruder / Scanner / Logger 导航)。
- 中上:工具栏(代理开关 / 端口 / 过滤);中:**虚拟化 `Table`** = HTTP history(method/host/path/status/len/time)。
- 下:请求 / 响应详情(双栏,只读文本;后续接 `InputState` 可编辑改包)。
- 主题:沿用 mage_ui 深色科技 + 玻璃;HTTP 状态 / 方法用 `Badge` 着色。
- 数据:先用 mock 行跑通布局;再接 `scry_storage` / `scry_proxy` channel 上真流量。

## 组件化原则
- 内核与 UI 严格分层:`scry_core` 只放共享类型,无 IO;IO / 平台能力各自归 `proxy/storage/ca`。
- 能在 `mage_ui` 框架层补的组件就补进 `mage_ui`(反哺),不在 `scry_app` 里堆一次性 UI。

## 安全与合规
- 仅用于授权测试 / 自有流量分析 / 学习逆向。根 CA 私钥仅存本机 `~/.scry/`,不入库、不外传。
