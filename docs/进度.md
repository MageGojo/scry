- **重构**:抓包模型从「按机制(代理 / 各网卡)」改为「**按抓什么**」+ **scry 自管流量源**(Burp 的杀手锏)。
- **四种抓包目标**(按推荐排序):
  - **T1 抓网站(内置浏览器)⭐ 默认/零配置**:scry 拉起内置 Chromium,喂 `--proxy-server=8888` + `--ignore-certificate-errors-spki-list=<scry CA SPKI>`(免装系统 CA、过 pinning)+ `--user-data-dir`(隔离)。QUIC 自动失效,不碰系统/QX,能抓 Google。
  - **T2 抓程序/命令(托管启动)零配置**:同一 launcher 拉起目标 + 注入 `HTTP(S)_PROXY` + `SSL_CERT_FILE`/`NODE_EXTRA_CA_CERTS`/`REQUESTS_CA_BUNDLE`/`SSLKEYLOGFILE`。覆盖 curl/Electron/Java/Python/Node,零风险。
  - **T3 抓整机/已运行 App(进阶)**:透明 pf rdr(⚠️ 已知硬限:macOS 抓不到本机自身流量,只能当网关)/ 被动嗅探 + pcapng(仅元数据)。
  - **T4 对接代理客户端(抓任意软件主力,进阶)**:sing-box 插件 / QX / Proxifier;scry 经 `upstream` 把流量交回客户端出网。
- **不变**:引擎(`scry_proxy` MITM + `upstream`)**零改动复用**;重构落在 `scry_app`(新增 launcher + Dashboard 四卡)+ 文档。
- **与零环境出包合流**:T1 需随包带 Chrome for Testing → 与四大能力最后一项「零环境出包(macOS .app)」一并交付。
- **落地序**:T1 内置浏览器 > T2 托管启动 > Dashboard 四卡改版 > T3/T4 并卡。**当前状态:T1 内置浏览器 / T2 托管启动 / Dashboard 四卡均已写码落地;T3 透明代理已删除;T4 沿用既有 sing-box 插件 / QX / Proxifier 对接。仅「零环境出包带 Chromium」未完成。**

## 不抢系统代理 + 解 HTTPS 的方案选型(已定)
- **环境**:墙内 + Quantumult X(TUN,系统唯一上网出口)。诊断结论:`/dev/bpf*` 仅 root;默认路由走 `utun6`(QX);**`en1` 直连出不去网**(墙内只有走 QX 能上网)。
- **选定:Burp 式「把客户端指向 scry」**(`docs/设计-解密抓包-与QX共存.md`)。浏览器(或 curl/SDK)代理设 `127.0.0.1:8888` → scry 解密 → 默认路由经 QX 出网。**系统代理 / QX 一行没动**(满足"不抢代理"),且**无回环**(QX 没有"目标→scry"规则)。Chrome `--proxy-server` 独立配置 / Firefox 手动代理即可。
- **否决:在 QX 里做 `HOST-SUFFIX,目标,scry` 链式**。QX **不支持 PROCESS-NAME**,无法排除 scry 自身上游 → 死循环;绑物理网卡绕 QX 在墙内又出不去网。两条死路。为此一度加的 `IP_BOUND_IF` 上游绑网卡(`scry_proxy::net` + socket2 + `ProxyConfig.upstream_iface`)已**全部回退**。
- **已评估搁置:macOS Network Extension(local 模式)**。最干净(按进程、不抢代理)但受限 entitlement 需签名+公证 / 关 SIP+AMFI,工程过重;引擎泛型化 / `PrefixedStream` 也已回退。
- **headless 实证**:`scry_proxy` 跑 8888,curl `-x` 经 QX 默认路由实测 HTTP + HTTPS(MITM 解密)+ httpbin 共 4 条全部落盘(200),证明"客户端指向 scry"所需的 CONNECT+解密管线在本机可用。
