# 设计 · 抓包方案重构(以「抓什么」为中心 + scry 自管流量源)

> 本文是对 scry 抓包方案的一次**统一重构**。把过去散落在 `设计-抓包源选择.md` /
> `设计-解密抓包-与QX共存.md` 及多轮排障里的经验,收敛成一套**默认零配置、优雅降级**的模型。
> 旧文档不删,作为机制细节参考;本文为新的总纲。

---

## 1. 为什么要重构(现状痛点)

现有抓包能力其实很全(MITM 代理 / 透明 pf rdr / 被动嗅探 / upstream 上游 / sing-box 插件 /
Proxifier),但**组织方式是「按机制」**(仪表盘列「代理 + 各网卡」),而**每一种默认机制都依赖脆弱的外部管线**:

- 代理模式 → 要用户去配 Proxifier 规则 / 浏览器代理 / sing-box 插件 onGenerate;
- 任何一处管线断(Proxifier 漏进程、sing-box 触发器被剔、系统没装 CA、QX fake-ip 丢 SNI、Chrome 走 QUIC 绕过)→ 直接「抓不到包」。
- 历次「抓不到包」事故,根因几乎都在**这层外部管线**,而不在 scry 的 MITM 引擎(引擎早已 headless 实证可用)。

**用户诉求**(反复出现):要 Burp / Wireshark 那样**开箱即用**,反感配 Proxifier、改系统代理、改 Chrome。

**重构主张**:抓包模型从「用哪个机制」改为「**抓什么**」;默认路径**由 scry 自己掌控流量源的启动**
(这正是 Burp 的杀手锏),把外部管线从「默认必经之路」降级为「进阶可选项」。

---

## 1.5 内核选型定论(联网核实 Burp / mitmproxy / Wireshark)

**仿 Burp 渗透工具的内核必须是 TLS 终止式 MITM 代理(L7),不能用 Wireshark/libpcap 被动嗅探当内核。** 三条硬伤:
1. **解不了现代 HTTPS**:TLS 1.3 前向保密,私钥也解不开;被动嗅探只能靠 `SSLKEYLOGFILE`(仅限自控且愿吐密钥的客户端,原生 App/C2 无效)。
2. **只读,改不了包**:Repeater/Intruder/Scanner/Intercept 都要在线改写;嗅探看到包时已发出,物理上无法拦截/修改。
3. **裸包 TCP 重组又苦又脏**:做代理直接读干净 HTTP 流,投入产出比远胜重组 libpcap 帧。

> Burp / mitmproxy / Charles / Fiddler / Reqable **无一**用 libpcap 当内核。scry 现有 `scry_proxy::mitm` 方向正确,
> 重心放在 HTTP/2·WS·gRPC + 改包断点 + 三件套,而非抓包重组。Wireshark 模式(`scry_sniff`)仅作**辅助降级**(pcapng/元数据/兜底),不扩张。

引流方式(mitmproxy 模式设计为参照):自启客户端(regular+scry 拉起)是默认;transparent(pf rdr)**mitmproxy 官方已弃用**,改推 WireGuard/local-capture。
macOS 上 pf rdr **抓不到本机自身流量**(只能当网关)——故 **scry 已删除 T3 透明代理(pf)**;抓本机任意已运行软件走 T4(代理客户端)或将来的 NetworkExtension。

## 2. 核心原则

1. **零外部依赖优先**:默认路径不需要 Proxifier、不改系统代理、不改日常 Chrome、不装系统 CA。
2. **以「抓什么」组织**:用户选目标(网站 / 程序 / 整机 / 代理客户端),scry 自动挑机制。
3. **统一汇聚到 MITM 引擎**:所有路径最终把流量喂到同一个 `scry_proxy` MITM(已成熟,零改动复用)。
4. **统一回流(upstream)**:scry 解密后经 `scry_proxy::upstream` 交回 QX / sing-box 出网(墙内),
   目标域名交上游远程解析 → 根治「墙内直连出不去」+「fake-ip 丢真实 SNI」。
5. **优雅降级,绝不断网**:能解密就 MITM 解密;解不开(pinning / MMTLS)就被动留元数据(sniff / pcapng),全程不阻断用户网络。

## 3. 四种「抓包目标」(Capture Target)

按推荐度 / 易用度排序。前两种零配置,是新的默认门面;后两种是进阶。

```
                       ┌───────────────────────────────────────────────┐
   T1 内置浏览器 ─┐    │            scry_proxy  MITM 引擎 (8888)         │
   T2 托管程序   ─┼──▶ │   动态签叶子证书 · 解密 · 落盘(save-first)·UI   │
   T3 透明/整机  ─┤    │                     │                          │
   T4 代理客户端 ─┘    │                     ▼ upstream                 │
                       │      QX / sing-box 本地入站 ──▶ 机场节点 ──▶ 目标 │
                       └───────────────────────────────────────────────┘
```

### T1 · 抓网站(内置浏览器)— 默认 / 推荐 / 零配置 ⭐
scry 自己拉起一个**内置 Chromium**(Chrome for Testing),启动参数:
- `--proxy-server=http://127.0.0.1:8888` —— 指向 scry MITM;
- `--ignore-certificate-errors-spki-list=<scry CA 的 SPKI base64>` —— **免装系统 CA,连 pinning 站都过**;
  - SPKI 算法:`openssl x509 -in ca.pem -pubkey -noout | openssl pkey -pubin -outform der | openssl dgst -sha256 -binary | openssl base64`;
- `--user-data-dir=<scry 独立 profile>` —— 隔离日常浏览器(**且此 flag 不配,上一条 SPKI 不生效**);
- `--no-first-run --no-default-browser-check`;
- 可选 `SSLKEYLOGFILE=~/.scry/keylog` —— 给被动嗅探 / pcapng 兜底解密。

要点:走 HTTP 代理时浏览器 **QUIC 自动失效**,不会绕过;**不碰系统 / 不碰 QX / 能抓 Google**。
这是 Burp 抓包的本质(让浏览器指向它,而非改系统),也是 scry 现在最该补的体验层。

### T2 · 抓某个程序 / 命令(托管启动)— 零配置,覆盖非浏览器
选一个可执行文件 / 命令行,scry **拉起它并注入环境变量**(与 T1 同一套 launcher,泛化):
- 代理:`HTTP_PROXY` / `HTTPS_PROXY` / `ALL_PROXY = http://127.0.0.1:8888`;
- CA 信任(按运行时分发):
  - curl / openssl → `SSL_CERT_FILE` / `CURL_CA_BUNDLE = ~/.scry/ca.pem`;
  - Node → `NODE_EXTRA_CA_CERTS`;
  - Python requests → `REQUESTS_CA_BUNDLE`;
  - Go → 认 `HTTPS_PROXY`,CA 用 `SSL_CERT_FILE`;
  - Java → `-Djavax.net.ssl.trustStore`(CA 需转 JKS,二期);
- 兜底:`SSLKEYLOGFILE`。

覆盖 curl / wget / Electron / Java / Python / Node / Go 等。**零风险**(只动自启进程的环境,不碰系统)。
局限:认环境变量代理 + 可换 CA 的程序才行;硬编码不走代理 / 自带 CA bundle 且不认 env 的(少数)→ 走 T3 / T4。

### T3 · 抓整机(被动嗅探)— 辅助降级,仅元数据
针对「不愿引流 / 解不开」的兜底观测。**透明代理 pf rdr 已删除**(macOS pf rdr 抓不到本机自身流量,
且 mitmproxy 官方已弃用 transparent 改推 WireGuard/local)。
- **被动嗅探**(`scry_sniff`,BPF):任意软件都能抓,但 HTTPS **只有元数据 + SNI**(不解密);
  配合 T1 / T2 设的 `SSLKEYLOGFILE` 可对「自启进程」做被动解密;另存 **pcapng** 给 Wireshark。
- **(将来)零配置抓任意原生 App** = macOS **NetworkExtension 系统扩展**(对标 mitmproxy local-capture):
  唯一干净的纯本机方案,但需 Apple 开发者签名 + 公证(工程重),无明确需求前不做。
- 抓「本机已运行、不认代理」的软件,现阶段走 **T4(代理客户端)** 最现实。

### T4 · 对接代理客户端(sing-box / QX / Proxifier)— 抓任意软件的现实主力
用户本来就开着 QX / sing-box 全局 TUN。让客户端把(选定的)流量当作上游交给 scry,
scry 解密后经 `upstream` 再交回客户端出网。**这是抓「任意软件」的现实主力**,但属于有外部依赖的进阶路径,不再当默认。
- **sing-box**:已写好 `plugin-scry-capture.js`(**生成 onGenerate 混入脚本**,非依赖插件触发器):
  出站 `scry`→8888 / 回流入站 `scry-upstream-in`→8899 / route 客户端入站 TCP→scry,**按 inbound 天然防回环**;
  CaptureCN 反选排除 CN / Private。已端到端验证。
- **Proxifier**:`Scry.ppx`(单 HTTPS/CONNECT 代理,scry peek 首字节自动分流明文 / TLS),按进程喂 + Direct 防回环。
- scry 抓包上游 = 设置页填 `socks5://127.0.0.1:8899`(切勿填 20122=QX 出口,否则回环)。

---

## 4. 统一回流(upstream)与防回环
- 所有路径的 scry MITM 上游统一走 `scry_proxy::upstream`(设置页配 / 或 `SCRY_UPSTREAM` 环境变量),
  指向 QX / sing-box 本地入站;目标以**域名字符串**交上游远程解析。
- 防回环靠「按 inbound / 进程 / 回流口区分」,**不依赖系统代理**:
  - T1 / T2:scry 是子进程,上游走默认路由 = QX,QX 无「目标→scry」规则 → 不回环;
  - T4 sing-box:回流入站 tag 与客户端入站不同 → 天然防环;
  - T4 Proxifier:给 scry 自身 + QX/Surge 加 Direct 规则。

## 8. 取舍与边界(诚实记录)
- T1 / T2 只能抓「scry 自己启动」的进程 —— 抓不了已经在跑的任意 App(那是 T4)。
- macOS pf rdr 抓不了本机自身流量(只能当网关)+ mitmproxy 官方已弃用 → **pf 透明代理已从代码删除**;抓本机任意已运行软件走 T4。
- SSL Pinning:T1 用 SPKI 白名单可过(且握手链已带 CA → 免装系统 CA);其它路径遇 pinning / 微信 MMTLS 仍解不开(任何 MITM 同理),降级到被动元数据。
- 引擎(`scry_proxy` MITM + upstream)**核心复用**;本次重构在 `scry_app`(新增 launcher + Dashboard 四卡)、`scry_ca`(SPKI)、`scry_proxy::mitm`(链附 CA)与文档 / UI 层。
